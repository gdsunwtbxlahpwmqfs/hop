# Hop Web 增量调试方法论

---

## 一、概述

本文档提供 Hop Web 开发环境下的增量调试方法论，指导开发人员快速验证代码修改是否正确部署到开发环境。

---

## 二、增量调试工作流程

### 2.1 流程概览

```
修改代码 → 增量编译 → 重新打包 → 重启服务器 → 功能验证
```

### 2.2 详细步骤

#### 步骤 1: 修改代码

根据需求修改相应的 Java 源文件。

#### 步骤 2: 增量编译 UI 模块

```bash
# 方式一：安装到本地仓库（推荐）
./mvnw install -pl ui -am -DskipTests -Dassemblies=false -q

# 方式二：仅编译（不安装到本地仓库）
./mvnw compile -pl ui -am -q
```

**参数说明**：
| 参数 | 说明 |
|------|------|
| `-pl ui` | 指定编译 ui 模块 |
| `-am` | 同时编译依赖模块 |
| `-DskipTests` | 跳过测试 |
| `-Dassemblies=false` | 跳过 assembly 打包 |
| `-q` | 静默模式 |

#### 步骤 3: 重新打包 Web Assembly

```bash
./mvnw package -pl assemblies/web -am -DskipTests -Dfast-build
```

**参数说明**：
| 参数 | 说明 |
|------|------|
| `-pl assemblies/web` | 指定打包 web assembly |
| `-Dfast-build` | 启用快速构建模式 |

#### 步骤 4: 重启开发服务器

```bash
# 停止现有容器（如果运行中）
docker compose -f docker-compose.dev.yml down --remove-orphans

# 启动开发服务器
./dev-scripts/start-web-dev.sh
```

**启动脚本会自动检测并重新提取**：
- `hop.war` 文件比 `webapp` 目录新 → 重新提取
- 核心组件 zip 文件更新 → 重新提取

#### 步骤 5: 功能验证

访问以下地址进行验证：

| 服务 | URL |
|------|-----|
| Web UI | http://localhost:8080/ui |
| REST API | http://localhost:8080/hop/ |
| Debug Port | 5005 (JDWP) |

## 三、增量更新原理

### 3.1 文件依赖关系

```
Java 源文件 (.java)
        ↓ 编译
  Class 文件 (.class)
        ↓ 打包
    JAR 文件 (.jar)
        ↓ 组装
    WAR 文件 (.war)
        ↓ 提取
  Webapp 目录
        ↓ 部署
   Docker 容器
```

### 3.2 增量检测机制

启动脚本 `start-web-dev.sh` 使用时间戳对比实现增量更新：

```bash
# 检查 webapp 是否需要重新提取
if [ -z "$(ls -A "$WEBAPP_DIR" 2>/dev/null)" ] || [ "assemblies/web/target/hop.war" -nt "$WEBAPP_DIR" ]; then
  rm -rf "$WEBAPP_DIR"
  unzip -q assemblies/web/target/hop.war -d "$WEBAPP_DIR"
fi
```

**检测条件**：
1. `webapp` 目录为空或不存在
2. `hop.war` 文件比 `webapp` 目录更新

## 四、关键文件路径

| 文件/目录 | 路径 | 说明 |
|-----------|------|------|
| UI 源码 | `ui/src/main/java/` | Hop GUI 源代码 |
| 编译输出 | `ui/target/classes/` | 编译后的 class 文件 |
| UI JAR | `ui/target/hop-ui-*.jar` | 打包后的 JAR 文件 |
| Web WAR | `assemblies/web/target/hop.war` | Web 应用 WAR 包 |
| 提取目录 | `assemblies/web/target/webapp/` | 解压后的 Web 应用 |
| 启动脚本 | `dev-scripts/start-web-dev.sh` | 开发服务器启动脚本 |
| Docker 配置 | `docker-compose.dev.yml` | Docker Compose 配置 |

## 五、常见问题排查

### 5.1 修改未生效

**可能原因**：
1. 未执行 `mvn install`，导致依赖模块未更新
2. WAR 包未重新打包
3. `webapp` 目录缓存未清理

**解决方案**：
```bash
# 清理并重新构建
./mvnw clean install -pl ui -am -DskipTests -Dassemblies=false
./mvnw package -pl assemblies/web -am -DskipTests -Dfast-build

# 手动清理缓存
rm -rf assemblies/web/target/webapp
```

### 5.2 编译失败

**排查步骤**：
```bash
# 查看详细错误信息
./mvnw compile -pl ui -am 2>&1 | grep -A 10 "error"

# 检查依赖问题
./mvnw dependency:tree -pl ui
```

### 5.3 服务器启动失败

**排查步骤**：
```bash
# 检查容器日志
docker logs hop-web-dev

# 检查端口占用
lsof -i :8080
lsof -i :5005

# 重新构建并启动
./mvnw clean package -pl assemblies/web -am -DskipTests -Dfast-build
./dev-scripts/start-web-dev.sh
```

### 5.4 调试连接失败

**检查项**：
1. 确保服务器已启动（端口 5005 开放）
2. 检查防火墙设置
3. 确认 JDWP 配置正确

## 六、最佳实践

### 6.1 开发流程建议

1. **使用增量编译**：修改代码后使用 `-pl ui -am` 仅编译相关模块
2. **保持服务器运行**：开发过程中保持 Docker 容器运行，只需重新打包
3. **利用调试端口**：使用 IDE 远程调试功能（端口 5005）

### 6.2 性能优化

```bash
# 使用并行编译加速
./mvnw install -pl ui -am -DskipTests -Dassemblies=false -T 1C

# 使用缓存加速
./mvnw install -pl ui -am -DskipTests -Dassemblies=false -q
```

### 6.3 日志查看

```bash
# 查看服务器日志
tail -f assemblies/web/target/webapp/WEB-INF/logs/*.log

# 查看 Docker 容器日志
docker logs -f hop-web-dev
```
