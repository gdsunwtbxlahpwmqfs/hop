# Qi Hop 离线部署说明书

> 本说明书面向**无外网访问**的生产环境，系统化阐述 Qi Hop（Hop Web / Hop REST）从打包、传输、部署、启动到验证的完整流程。
>
> 配套 **8 个自动化脚本**（依赖下载、打包、部署、启动/停止、验证、卸载、**LLM 助手独立部署**），帮助技术人员在无网络环境下独立完成部署。
>
> **架构分层**：
> - **主部署（离线）**：Hop Web 本体，通过 `hop-deploy.sh` 部署到主机原生进程，完全无网络
> - **LLM 助手（可选，需临时联网）**：LiteLLM 代理 + Qdrant 向量库，脚本随主包分发（`llm/` 目录），通过 `hop-deploy-llm.sh` 独立部署到 Docker，部署时需临时联网拉取镜像，运行时仅 LLM API 调用需要公网

## 1. 部署包打包规范

### 1.1 打包内容

离线部署包须包含以下完整组件，缺一不可（除标注"可选"外）：

| 类别  | 组件                             | 必需 | 来源                                                                    |
| --- | ------------------------------ | -- | --------------------------------------------------------------------- |
| 运行时 | JDK 21 (Linux x64)             | ✅  | [Adoptium Temurin](https://adoptium.net/temurin/releases/?version=21) |
| 容器  | Apache Tomcat 10.1.x           | ✅  | <https://tomcat.apache.org/download-10>                               |
| 应用  | `hop.war`                      | ✅  | `assemblies/web/target/hop.war`                                       |
| 核心库 | `hop-client-*.zip`             | ✅  | `assemblies/client/target/`（提供 hop-core、hop-engine 等）                 |
| 插件  | `hop-assemblies-plugins-*.zip` | 推荐 | `assemblies/plugins/target/`                                          |
| 数据库 | JDBC 驱动 `.jar`                 | 可选 | 各数据库厂商官网                                                              |
| 脚本  | `deploy-scripts/*.sh`          | ✅  | 本仓库 `deploy-scripts/`（共 8 个：package/download-deps/deploy/start/stop/verify/uninstall/**deploy-llm**）                                                 |
| 文档  | 本说明书                           | ✅  | 本文件                                                                   |

### 1.3 压缩格式

- **统一压缩格式**：`tar.gz`（保留 Linux 权限位、符号链接）
- **命名规范**：`hop-offline-<版本>-<构建时间>.tar.gz`
- **示例**：`hop-offline-2.19.0-SNAPSHOT-20260719-143012.tar.gz`

### 1.4 版本标识

`MANIFEST.txt` 文件记录以下元数据，部署脚本会读取并写入启动日志：

```text
# Qi Hop 离线部署包清单
hop_version=2.19.0-SNAPSHOT
build_date=20260719-143012
target_platform=linux
package_name=hop-offline-2.19.0-SNAPSHOT-20260719-143012
generated_by=deploy-scripts/hop-package.sh

# 包含组件
hop_war=hop.war
hop_client=hop-client-2.19.0-SNAPSHOT.zip
hop_plugins=hop-assemblies-plugins-2.19.0-SNAPSHOT.zip

tomcat=apache-tomcat-10.1.24.tar.gz
jdbc_drivers=5

# 部署脚本清单（写入 SHA256SUMS 一并校验）
deploy_scripts=hop-deploy.sh,hop-start.sh,hop-stop.sh,hop-verify.sh,hop-uninstall.sh
```

### 1.5 打包操作（在有网络的机器上）

#### 1.5.1 自动打包（推荐）

完整的离线打包流程包含两步：**下载依赖**（仅首次或升级 JDK/Tomcat 时需要）+ **打包**。

```bash
# Step 1：下载 JDK 21 / Tomcat 10.1 离线包（在有网络的机器上执行一次）
./deploy-scripts/hop-download-deps.sh                       # 默认 linux 平台
./deploy-scripts/hop-download-deps.sh --platform linux      # 显式指定
./deploy-scripts/hop-download-deps.sh --list                # 列出支持的版本
./deploy-scripts/hop-download-deps.sh --verify              # 仅校验已下载文件
# 产物：deploy-scripts/downloads/{jdk,tomcat}/*.tar.gz（含 SHA256 内嵌校验）

# Step 2：打包离线部署包
cd /path/to/hop

# 一键打包：构建 + 组织 + 校验 + 压缩
./deploy-scripts/hop-package.sh

# 跳过 Maven 构建（已有产物时）
./deploy-scripts/hop-package.sh --skip-build

# 指定目标平台（默认 linux）
./deploy-scripts/hop-package.sh --target linux

# 同时指定 JDK/Tomcat 离线包路径（推荐配合 Step 1 的产物）
./deploy-scripts/hop-package.sh \
    --jdk deploy-scripts/downloads/jdk/jdk-21_linux-x64_bin.tar.gz \
    --tomcat deploy-scripts/downloads/tomcat/apache-tomcat-10.1.24.tar.gz

# 指定输出目录
./deploy-scripts/hop-package.sh --output /tmp

# 覆盖版本号（默认从 pom.xml <revision> 读取）
./deploy-scripts/hop-package.sh --version 2.19.0-SNAPSHOT
```

#### 1.5.2 手动打包

```bash
# 1. 构建 Hop
./mvnw clean install -Pfast-build -DskipTests \
    -Dswt.artifactId=org.eclipse.swt.gtk.linux.x86_64 \
    -Denv=linux -P'assemblies'

# 2. 组织目录（参考 1.2 节结构）

# 3. 生成校验文件
cd hop-offline-xxx/
find . -type f ! -name 'SHA256SUMS' -exec sha256sum {} \; > SHA256SUMS

# 4. 打包
cd ..
tar -czf hop-offline-xxx.tar.gz hop-offline-xxx/
sha256sum hop-offline-xxx.tar.gz > hop-offline-xxx.tar.gz.sha256
```

| 资源        | 最低    | 推荐    | 生产           |
| --------- | ----- | ----- | ------------ |

> \* **内存说明**：`hop-deploy.sh` 默认 `-Xms16g -Xmx16g`（对齐生产负载）。若服务器内存 ≤ 8 GB，部署时**必须**通过 `--heap` 参数降低堆大小，例如 `--heap 4g`；否则 JVM 启动会因无法预留堆而失败。验证脚本（`hop-verify.sh`）的内存检查阈值为 ≥ 4096 MB。

### 2.4 库文件与版本兼容性

| 库                                        | 来自         | 兼容说明                                         |
| ---------------------------------------- | ---------- | -------------------------------------------- |
| `hop-core-*.jar`                         | hop-client | 核心 API，**必须存在**于 `WEB-INF/lib`               |
| `hop-engine-*.jar`                       | hop-client | Pipeline/Workflow 执行引擎                       |
| `hop-ui-rap-*.jar`                       | hop.war    | RAP Web UI 入口                                |
| `org.eclipse.rap.rwt-*.jar`              | hop.war    | RAP 运行时（RAP 模式下唯一的 SWT 实现）                  |
| `org.eclipse.swt.*`（桌面实现）                | hop-client | **RAP 模式下必须移除**（cocoa/gtk.linux/win32 全部），由 `hop-deploy.sh` 的 `clean_rap_conflicts` 自动清理 |
| JDBC 驱动 jar                              | 各厂商        | 放置于 `${CATALINA_BASE}/jdbc-drivers/`（如 `/opt/hop/tomcat-run/jdbc-drivers/`） |

**关键兼容性规则**：

1. **JDK 21 是硬性要求**：Hop 编译目标为 Java 21，低版本会报 `UnsupportedClassVersionError`。
2. **Tomcat 10.x 是硬性要求**：Hop 使用 `jakarta.*` 命名空间（Jakarta EE 9+），Tomcat 9 及以下（`javax.*`）**完全不兼容**。
3. **hop.war 不含核心库**：必须从 `hop-client-*.zip` 提取 `hop/lib/core/*.jar`、`hop/lib/beam/*.jar` 到 `WEB-INF/lib`。
4. **RAP 模式清理桌面 SWT**：`hop-deploy.sh` 会移除 `WEB-INF/lib` 下的所有 `org.eclipse.swt.{cocoa,gtk.linux,win32}*`，因为 RAP 唯一依赖 `org.eclipse.rap.rwt` 作为 SWT 实现；保留桌面 SWT 会导致 `Invalid thread access` 或 native 库加载失败。
5. **移除 hop-ui-rcp 及其 runtime 依赖**：`hop-ui-rcp*`、`org.eclipse.tm4e.core*`、`org.eclipse.jface.text*` 会被一并清理，避免与 `hop-ui-rap` 的 `TextSizeUtilFacadeImpl` SPI 冲突。

### 2.5 Dockerfile 环境依赖对齐（参考 `docker/web.Dockerfile`）

部署脚本 `hop-deploy.sh` 的环境变量与 `docker/web.Dockerfile` 完全对齐，便于在容器化与主机部署之间无障碍切换。

> **目录布局重大调整**：自 v1.1 起，主机部署采用 **CATALINA_HOME / CATALINA_BASE 分离** 结构（对齐 `dev-scripts/start-web-dev.sh` 的 `tomcat/` + `tomcat-run/` 模式）：
>
> - `CATALINA_HOME=${INSTALL_BASE}/tomcat`（Tomcat 软件目录，**只读**，多实例共享）
> - `CATALINA_BASE=${INSTALL_BASE}/tomcat-run`（运行实例，**可写**，单实例默认名）
>   - 多实例：`CATALINA_BASE=${INSTALL_BASE}/tomcat-run-${INSTANCE_NAME}`
>
> Hop 数据目录（audit/config/plugins/jdbc-drivers/rwt-resources）全部内聚到 `CATALINA_BASE` 下，便于多实例隔离与升级时保留实例数据。

#### 2.5.4 Dockerfile 关键 RUN 步骤对齐

| Dockerfile RUN                                                  | hop-deploy.sh 对应                                                                  | 说明                                |
| --------------------------------------------------------------- | ---------------------------------------------------------------------------------- | --------------------------------- |
| `rm -rf webapps/*`                                              | `rm -rf "${CATALINA_HOME}/webapps/"*`                                              | 清理 CATALINA_HOME 默认 webapp        |
| `mkdir webapps/ROOT`                                            | `mkdir -p "$WEBAPP_DIR"`（位于 `${CATALINA_BASE}/webapps/ROOT`）                     | 创建 ROOT（在 CATALINA_BASE 实例目录）    |
| `mkdir "${HOP_AUDIT_FOLDER}"`                                   | `mkdir -p "${CATALINA_BASE}/audit"`                                                | 审计目录（内聚到 CATALINA_BASE）           |
| `COPY webapp/ -> ROOT/`                                         | `unzip hop.war -> ROOT`                                                            | 部署 Web 应用                         |
| `COPY hop/lib/core -> WEB-INF/lib`                              | `cp hop/lib/core/*.jar -> WEB-INF/lib`                                             | 核心库                               |
| `COPY hop/lib/beam -> WEB-INF/lib`                              | `cp hop/lib/beam/*.jar -> WEB-INF/lib`                                             | Beam 引擎库                          |
| `COPY hop/plugins -> plugins`                                   | `mv plugins/* -> ${CATALINA_BASE}/plugins`                                         | 插件部署（实例目录）                        |
| `COPY hop/lib/jdbc -> jdbc-drivers`                             | `cp jdbc-drivers/*.jar -> ${CATALINA_BASE}/jdbc-drivers`                           | JDBC 驱动                           |
| `sed 's/config\/projects/${HOP_CONFIG_FOLDER}\/projects/g'`     | `sed "s\|config/projects\|${HOP_CONFIG_FOLDER}/projects\|g"`                       | hop-config.json 路径修正              |

| `sed 's&lib/core/*&../../lib/*:WEB-INF/lib/*:lib/core/*&g'` × 6 | 同（5 个 CLI 脚本循环）                                                                   | classpath 修正                      |
| `chmod +x ROOT/*.sh`                                            | `chmod +x "$WEBAPP_DIR"/*.sh`                                                      | CLI 脚本可执行                         |
| `chown -R hop:hop /usr/local/tomcat`                            | `chown -R ${HOP_USER}:${HOP_GROUP} ${CATALINA_BASE}` + `${CATALINA_HOME}/lib/swt` | 权限（实例目录 + SWT 子目录）                |

## 5. 系统启动程序说明

### 5.1 启动命令（使用配套脚本）

```bash
# 启动服务（后台运行，自动健康检查，等待 /hop/status/ 返回）
sudo bash /opt/deploy/hop-offline-*/scripts/hop-start.sh start

# 停止服务（等价于 hop-stop.sh，优雅退出 + 强制兜底）

# 或：sudo bash /opt/deploy/hop-offline-*/scripts/hop-stop.sh

# 重启服务
sudo bash /opt/deploy/hop-offline-*/scripts/hop-start.sh restart

# 查看状态（PID/端口/REST 接口/CATALINA_BASE）
sudo bash /opt/deploy/hop-offline-*/scripts/hop-start.sh status

# 健康检查（探活 REST API，输出响应耗时）
sudo bash /opt/deploy/hop-offline-*/scripts/hop-start.sh health

# 前台运行（调试用，输出到终端）
sudo bash /opt/deploy/hop-offline-*/scripts/hop-start.sh foreground
# 简写：fg

# 查看实时日志
sudo bash /opt/deploy/hop-offline-*/scripts/hop-start.sh log
# 简写：logs

# 多实例：通过 --instance 或环境变量指定实例名
sudo bash /opt/deploy/hop-offline-*/scripts/hop-start.sh --instance bi status
INSTANCE_NAME=bi sudo -E bash /opt/deploy/hop-offline-*/scripts/hop-start.sh status
```

> **systemd 优先策略**：`hop-start.sh` 在 root 权限且检测到 `${SERVICE_NAME}.service` 已注册时，自动走 `systemctl`；否则回退到 `startup.sh`/`shutdown.sh` 直接调用，并显式 `export CATALINA_BASE` 以正确分离实例。

### 5.2 原生 Tomcat 命令（备选）

> 由于采用了 CATALINA\_HOME / CATALINA\_BASE 分离结构，**必须显式设置 CATALINA\_BASE** 才能定位实例；否则 Tomcat 会以 CATALINA\_HOME 作为 base，导致日志、webapps、setenv.sh 全部错位。

```bash
# 关键环境变量（每次执行前导出，或写入 /home/hop/.bashrc）
export JAVA_HOME=/opt/hop/jdk21
export CATALINA_HOME=/opt/hop/tomcat           # Tomcat 软件目录（只读）
export CATALINA_BASE=/opt/hop/tomcat-run       # 运行实例目录（可写）

# 启动
sudo su - hop -c "export JAVA_HOME=$JAVA_HOME CATALINA_HOME=$CATALINA_HOME CATALINA_BASE=$CATALINA_BASE; $CATALINA_HOME/bin/startup.sh"

# 停止
sudo su - hop -c "export JAVA_HOME=$JAVA_HOME CATALINA_HOME=$CATALINA_HOME CATALINA_BASE=$CATALINA_BASE; $CATALINA_HOME/bin/shutdown.sh"

# 前台运行（调试）
sudo su - hop -c "export JAVA_HOME=$JAVA_HOME CATALINA_HOME=$CATALINA_HOME CATALINA_BASE=$CATALINA_BASE; $CATALINA_HOME/bin/catalina.sh run"
```

### 5.3 参数配置方法

启动参数集中存储于 `${CATALINA_BASE}/bin/setenv.sh`（Tomcat 启动时自动加载），由 `hop-deploy.sh` 根据 `--heap` / `--heap-mode` / `--log-level` 等参数自动生成。

```bash
# JVM 堆内存（fixed 模式默认 16g，按服务器内存调整）
CATALINA_OPTS="${CATALINA_OPTS} -Xms16g -Xmx16g"

# GC 策略（fixed 模式默认启用；aggressive 模式由 -XX:+AggressiveHeap 接管）
CATALINA_OPTS="${CATALINA_OPTS} -XX:+UseG1GC -XX:MaxGCPauseMillis=200"

# OOM 时自动 dump（dump 文件位于 ${CATALINA_BASE}/logs/）
CATALINA_OPTS="${CATALINA_OPTS} -XX:+HeapDumpOnOutOfMemoryError"
CATALINA_OPTS="${CATALINA_OPTS} -XX:HeapDumpPath=${CATALINA_BASE}/logs/"

# 远程调试（生产环境禁用）
# CATALINA_OPTS="${CATALINA_OPTS} -agentlib:jdwp=transport=dt_socket,server=y,suspend=n,address=5005"
```

> **修改 setenv.sh 的正确姿势**：`hop-deploy.sh` 每次执行都会**覆盖** `setenv.sh`（重新生成）。如需固化自定义参数，推荐方式：
>
> 1. **首选**：通过 `--heap` / `--heap-mode` / `--log-level` 等 CLI 参数传递（下次部署自动应用）
> 2. **运行时调整**：部署完成后直接编辑 `${CATALINA_BASE}/bin/setenv.sh`（立即生效，但下次执行 `hop-deploy.sh` 会被覆盖，需重新编辑）
> 3. **环境变量注入**：通过 systemd unit 的 `Environment=` 或 `hop-deploy.sh` 的 `HOP_PROJECT_*` 等环境变量传递
> 4. **彻底定制**：修改 `deploy-scripts/hop-deploy.sh` 的 `generate_setenv` 函数模板（适合团队级固化）

修改后须重启服务生效：

```bash
sudo bash /opt/deploy/hop-offline-*/scripts/hop-start.sh restart
# 或：sudo systemctl restart qi-hop-web
```

```bash
[Service]
Type=forking
PIDFile=/opt/hop/tomcat-run/temp/tomcat.pid
User=hop
Group=hop
Environment="JAVA_HOME=/opt/hop/jdk21"
Environment="JRE_HOME=/opt/hop/jdk21"
Environment="CATALINA_PID=/opt/hop/tomcat-run/temp/tomcat.pid"
Environment="CATALINA_HOME=/opt/hop/tomcat"
Environment="CATALINA_BASE=/opt/hop/tomcat-run"
Environment="CATALINA_OUT=/opt/hop/tomcat-run/logs/catalina.out"
ExecStart=/opt/hop/tomcat/bin/startup.sh
ExecStop=/opt/hop/tomcat/bin/shutdown.sh
RestartSec=10
Restart=on-failure
LimitNOFILE=65536
LimitNPROC=65536
WorkingDirectory=/opt/hop/tomcat-run

[Install]
WantedBy=multi-user.target
EOF
```

> **关键差异**：相比旧版（单 tomcat 目录）模板，新版必须**显式声明 `CATALINA_HOME` 与 `CATALINA_BASE` 两个变量**，且 `PIDFile` / `CATALINA_OUT` / `WorkingDirectory` 都指向 `${CATALINA_BASE}`；`ExecStart`/`ExecStop` 仍指向 `${CATALINA_HOME}/bin/`（startup.sh 会读取 `CATALINA_BASE` 环境变量定位实例）。

## 8. 常见问题排查指南

### 8.1 启动报 `NoClassDefFoundError: HopException`

**原因**：`hop.war` 不含核心库（hop-core/hop-engine），`WEB-INF/lib` 缺失核心 JAR。

**解决**：

```bash
# 检查核心库是否存在（路径在 CATALINA_BASE 下）
ls /opt/hop/tomcat-run/webapps/ROOT/WEB-INF/lib/hop-core*.jar
ls /opt/hop/tomcat-run/webapps/ROOT/WEB-INF/lib/hop-engine*.jar

# 缺失则从 hop-client 包补充
CLIENT_ZIP=/opt/deploy/hop-offline-*/hop/hop-client-*.zip
unzip -q "$CLIENT_ZIP" "hop/lib/core/*" -d /tmp/hop-fix
cp /tmp/hop-fix/hop/lib/core/*.jar /opt/hop/tomcat-run/webapps/ROOT/WEB-INF/lib/
chown -R hop:hop /opt/hop/tomcat-run/webapps/ROOT/WEB-INF/lib/
sudo systemctl restart qi-hop-web
```

### 8.2 启动报 `ClassNotFoundException: ... SWT ...`

**原因**：RAP 模式下应只保留 `org.eclipse.rap.rwt` 作为 SWT 实现；若残留桌面 SWT（cocoa/gtk/win32），会导致 `Invalid thread access` 或 native 库加载失败。

**说明**：`hop-deploy.sh` 的 `clean_rap_conflicts` 函数会自动清理 `org.eclipse.swt.{cocoa,gtk.linux,win32}*` 等冲突 jar，正常部署**不应出现此问题**。如手动操作过 `WEB-INF/lib/`，可重新执行清理：

```bash
WEBAPP_LIB=/opt/hop/tomcat-run/webapps/ROOT/WEB-INF/lib
# 移除所有桌面 SWT 实现（RAP 唯一依赖是 org.eclipse.rap.rwt）
rm -f "$WEBAPP_LIB"/org.eclipse.swt.cocoa.*
rm -f "$WEBAPP_LIB"/org.eclipse.swt.gtk.linux.*
rm -f "$WEBAPP_LIB"/org.eclipse.swt.win32.*
# 确认 RAP RWT 仍在
ls "$WEBAPP_LIB"/org.eclipse.rap.rwt-*.jar
```

### 8.3 启动报 `TextSizeUtilFacadeImpl` 冲突

**原因**：`hop-ui-rcp` fragment 与 `hop-ui-rap` 重复提供同一 SPI；同时 `org.eclipse.tm4e.core`、`org.eclipse.jface.text` 作为 RCP runtime 依赖在 RAP 模式下冗余。

**说明**：`hop-deploy.sh` 的 `clean_rap_conflicts` 已自动清理上述 jar（对齐 `start-hop-web.sh` 纯 RAP 策略 B）。如部署后仍出现该异常，可手动补一次清理：

```bash
WEBAPP_LIB=/opt/hop/tomcat-run/webapps/ROOT/WEB-INF/lib
rm -f "$WEBAPP_LIB"/hop-ui-rcp*
rm -f "$WEBAPP_LIB"/org.eclipse.tm4e.core*
rm -f "$WEBAPP_LIB"/org.eclipse.jface.text*
sudo systemctl restart qi-hop-web
```

### 8.4 页面白屏 / RWT 资源加载失败

**原因**：RAP 资源缓存目录（`${CATALINA_BASE}/rwt-resources`）权限不足。

**解决**：

```bash
sudo mkdir -p /opt/hop/tomcat-run/rwt-resources
sudo chown -R hop:hop /opt/hop/tomcat-run/rwt-resources
sudo chmod 755 /opt/hop/tomcat-run/rwt-resources

# 确认 setenv.sh 中已配置 -Dorg.eclipse.rap.rwt.resourceLocation
grep "resourceLocation" /opt/hop/tomcat-run/bin/setenv.sh
```

> RWT 启动时会从 WAR classpath 自填充 xterm/monaco 等静态资源到该目录，**无需预拷贝**。

### 8.5 端口 8080 被占用

```bash
# 查看占用进程
ss -tlnp | grep 8080
lsof -i:8080

# 方案一：停止占用进程
sudo kill -9 <PID>

# 方案二：修改 Hop Web 端口（注意 server.xml 在 CATALINA_BASE/conf 下）
sudo sed -i 's/port="8080"/port="9090"/' /opt/hop/tomcat-run/conf/server.xml
# 多实例部署更推荐用 --port 参数重新部署一个新实例：
#   sudo bash scripts/hop-deploy.sh --instance bi --port 9090
```

### 8.6 插件未加载 / 数据库连接不可用

**排查步骤**：

```bash
# 确认目录有内容（路径在 CATALINA_BASE 下）
ls /opt/hop/tomcat-run/plugins/ | head -20
ls /opt/hop/tomcat-run/jdbc-drivers/

# 确认 setenv.sh 中环境变量已正确生成
grep -E "HOP_PLUGIN|HOP_SHARED_JDBC" /opt/hop/tomcat-run/bin/setenv.sh

# 确认运行中进程实际继承的环境变量（需 root）
pid=$(pgrep -f "catalina.base=/opt/hop/tomcat-run" | head -1)
sudo tr '\0' '\n' < /proc/$pid/environ | grep -E "HOP_PLUGIN|HOP_SHARED_JDBC"

# 检查 catalina.out 中插件加载记录
grep -i "plugin" /opt/hop/tomcat-run/logs/catalina.out | head -20
grep -i "jdbc" /opt/hop/tomcat-run/logs/catalina.out | head -20
```

### 8.7 内存溢出（OutOfMemoryError）

```bash
# 方式一（强烈推荐）：用 --heap 参数重新部署，避免手改 setenv.sh 被下次部署覆盖
sudo bash /opt/deploy/hop-offline-*/scripts/hop-deploy.sh --heap 32g

# 方式二：直接编辑 setenv.sh（注意：会被下次 hop-deploy.sh 覆盖）
# 查看当前堆配置
grep -E "^CATALINA_OPTS=.*-Xm[sx]" /opt/hop/tomcat-run/bin/setenv.sh
# 按实际值替换（示例：从 16g 升到 32g）
sudo sed -i -E 's/-Xmx[0-9]+[gm]/-Xmx32g/' /opt/hop/tomcat-run/bin/setenv.sh
sudo sed -i -E 's/-Xms[0-9]+[gm]/-Xms32g/' /opt/hop/tomcat-run/bin/setenv.sh

# 重启
sudo systemctl restart qi-hop-web
```

```bash
# CentOS/RHEL（firewalld）
sudo firewall-cmd --permanent --add-port=8080/tcp
sudo firewall-cmd --reload
sudo firewall-cmd --list-ports
```

### 8.9 Tomcat 启动很慢（超过 2 分钟）

**原因**：Tomcat `SessionIdGenerator` 使用 `/dev/random`，熵不足时阻塞。

**解决**：`hop-deploy.sh` 生成的 `setenv.sh` 已默认包含此项，确认未被删除：

```bash
grep "java.security.egd" /opt/hop/tomcat-run/bin/setenv.sh
# 应输出：CATALINA_OPTS="${CATALINA_OPTS} -Djava.security.egd=file:/dev/./urandom"
```

### 8.10 服务无法启动（端口被旧进程占用）

```bash
# 查找残留进程（多实例必须用 catalina.base 精确匹配，避免误杀其他实例）
pgrep -f "catalina.base=/opt/hop/tomcat-run"
# 强制清理
sudo pkill -9 -f "catalina.base=/opt/hop/tomcat-run"
# 清理 PID 文件（PID 文件在 CATALINA_BASE/temp 下）
sudo rm -f /opt/hop/tomcat-run/temp/tomcat.pid
# 重新启动
sudo systemctl start qi-hop-web
```

### 8.11 升级流程

> **多实例升级**：每个实例独立 CATALINA_BASE，可单独升级；CATALINA_HOME（Tomcat 软件）和 JDK 为共享资源，仅最后一个实例卸载后才会清理。
>
> **重要**：`hop-deploy.sh` 不会自动清理 `${CATALINA_BASE}/config/projects/`（用户项目数据）。升级前务必备份 `config/` 目录；若跨大版本升级，建议先卸载旧实例（`hop-uninstall.sh --yes`）再重新部署，避免旧版本残留 jar 干扰。

```bash
# 1. 停止服务
sudo systemctl stop qi-hop-web
# 或：sudo bash /opt/deploy/hop-offline-*/scripts/hop-stop.sh

# 2. 备份实例数据（CATALINA_BASE 内的配置/项目/审计）
sudo cp -r /opt/hop/tomcat-run/webapps/ROOT /opt/hop/tomcat-run/webapps/ROOT.bak.$(date +%Y%m%d)
sudo cp /opt/hop/tomcat-run/bin/setenv.sh /opt/hop/tomcat-run/bin/setenv.sh.bak.$(date +%Y%m%d)
sudo cp -r /opt/hop/tomcat-run/config /root/hop-config-backup-$(date +%Y%m%d)
sudo cp -r /opt/hop/tomcat-run/audit  /root/hop-audit-backup-$(date +%Y%m%d) 2>/dev/null || true

# 3a. 方式一：原地升级（保留 config/projects/，hop-deploy.sh 会跳过已存在的 conf/）
sudo bash /opt/deploy/hop-offline-<new-version>/scripts/hop-deploy.sh

# 3b. 方式二：彻底重装（推荐跨大版本时使用，避免旧 jar 残留）
sudo bash /opt/deploy/hop-offline-<old-version>/scripts/hop-uninstall.sh --yes
sudo bash /opt/deploy/hop-offline-<new-version>/scripts/hop-deploy.sh
# 然后手动恢复项目数据：
sudo cp -r /root/hop-config-backup-YYYYMMDD/projects /opt/hop/tomcat-run/config/
sudo chown -R hop:hop /opt/hop/tomcat-run/config

# 4. 启动
sudo systemctl start qi-hop-web

# 5. 验证
sudo bash /opt/deploy/hop-offline-<new-version>/scripts/hop-verify.sh
```

### 8.12 卸载流程

**推荐使用 `hop-uninstall.sh`**（自动处理 systemd 注销、进程清理、多实例共享资源保护）：

```bash
# 预览卸载计划（不实际执行）
sudo bash /opt/deploy/hop-offline-*/scripts/hop-uninstall.sh --dry-run

# 卸载当前实例（默认 tomcat-run/，会保留 CATALINA_HOME/JDK 等共享资源）
sudo bash /opt/deploy/hop-offline-*/scripts/hop-uninstall.sh --yes

# 卸载多实例中的指定实例
sudo bash /opt/deploy/hop-offline-*/scripts/hop-uninstall.sh --instance bi --yes

# 彻底清理：连同 CATALINA_HOME、JDK、用户、INSTALL_BASE 全部删除
# 仅在最后一个实例卸载时允许，否则拒绝执行
sudo bash /opt/deploy/hop-offline-*/scripts/hop-uninstall.sh --purge --yes
```

**手动卸载（不推荐，作为兜底参考）**：

```bash
# 1. 停止并禁用服务
sudo systemctl stop qi-hop-web
sudo systemctl disable qi-hop-web

# 2. 删除服务文件
sudo rm /etc/systemd/system/qi-hop-web.service
sudo systemctl daemon-reload

# 3. 兜底清理进程（多实例必须用 catalina.base 区分）
sudo pkill -9 -f "catalina.base=/opt/hop/tomcat-run"

# 4. 备份重要数据（项目、配置位于 CATALINA_BASE 下）
sudo cp -r /opt/hop/tomcat-run/config /root/hop-config-backup-$(date +%Y%m%d)

# 5. 删除实例目录（保留 CATALINA_HOME 与 JDK 以便其他实例复用）
sudo rm -rf /opt/hop/tomcat-run

# 6. 若是最后一个实例，再清理共享资源
sudo rm -rf /opt/hop/tomcat /opt/hop/jdk21
sudo rm -rf /opt/hop

# 7. 删除用户（可选）
sudo userdel hop     # 不带 -r，保留 /home/hop 数据
```

### 9.9 多实例部署（同机多 Hop Web）

依托 `CATALINA_HOME/BASE` 分离结构，可在同一台主机上部署多个独立 Hop Web 实例：

- **共享**：`CATALINA_HOME`（Tomcat 软件）、JDK（节省磁盘与内存复制开销）
- **独立**：`CATALINA_BASE`（webapps、config、plugins、logs、端口、systemd 服务名）

```bash
# 实例 1（默认）：tomcat-run/，端口 8080，服务 qi-hop-web
sudo bash scripts/hop-deploy.sh

# 实例 2：tomcat-run-bi/，端口 9090，服务 qi-hop-web-bi
sudo bash scripts/hop-deploy.sh --instance bi --port 9090

# 实例 3：tomcat-run-etl/，端口 9092，服务 qi-hop-web-etl，自定义堆
# 注意：相邻实例的 HTTP 端口至少错开 2（避免实例 N 的 AJP=HTTP_N+1 撞上实例 N+1 的 HTTP）
sudo bash scripts/hop-deploy.sh --instance etl --port 9092 --heap 8g
```

**端口自动错开**（多实例模式）：HTTP 端口取 `--port` 值；shutdown 端口 = HTTP - 75；AJP 端口 = HTTP + 1。例如 `--port 9090` → shutdown=9015、AJP=9091。

> ⚠️ **端口规划建议**：相邻实例的 HTTP 端口至少错开 **2**，否则实例 N 的 AJP 端口（HTTP_N+1）会与实例 N+1 的 HTTP 端口冲突。推荐用 `8080`、`9090`、`9092`、`9094`... 这样的偶数错开序列。

**多实例运维命令**：

```bash
# 状态查询
sudo bash scripts/hop-start.sh --instance bi status

# 停止指定实例
sudo bash scripts/hop-stop.sh --instance bi

# 验证指定实例
sudo bash scripts/hop-verify.sh --instance bi

# 卸载指定实例（保留共享的 CATALINA_HOME/JDK）
sudo bash scripts/hop-uninstall.sh --instance bi --yes
```

### 9.10 卸载脚本详解（`hop-uninstall.sh`）

卸载脚本分 5 个阶段，按顺序执行；多实例场景下会自动保护共享资源。

| 阶段       | 动作                                                          | `--purge` 影响                          |
| -------- | ----------------------------------------------------------- | -------------------------------------- |
| 1. systemd | 停止 + disable + 删除 unit 文件 + daemon-reload                    | 同                                      |
| 2. 进程清理   | SIGTERM 优雅退出 → 30s 超时 → SIGKILL；用 `catalina.base` 精确匹配实例    | 同                                      |
| 3. 实例目录   | 删除 `${CATALINA_BASE}`（强制校验 basename 必须为 `tomcat-run[-*]`）  | 同                                      |
| 4. 共享资源   | 检查剩余实例数；若 >0 保留 CATALINA\_HOME/JDK；若 =0 提示需 `--purge`        | 删除 `${CATALINA_HOME}` 与 `${JAVA_HOME}` |
| 5. 用户/根目录 | 跳过                                                          | 删除 `${INSTALL_BASE}`、`userdel hop`、`groupdel hop` |

**安全护栏**：

- `--purge` 在仍存在其他实例时**拒绝执行**，并列出待卸载的实例名
- `--dry-run` 仅打印动作，不实际执行（推荐首次执行先预览）
- `--yes` 跳过交互式确认（CI/CD 友好）
- 路径校验：删除前检查 basename 必须为 `tomcat-run` 或 `tomcat-run-*`，否则拒绝
- `userdel` 不带 `-r`：保留 `/home/hop` 用户数据，需手动清理

**完整参数列表**：`--base`、`--instance`、`--service`、`--user`、`--group`、`--purge`、`--yes`/`-y`、`--dry-run`。

```bash
# 预览卸载计划
sudo bash scripts/hop-uninstall.sh --dry-run

# 交互式卸载（输入 yes 确认）
sudo bash scripts/hop-uninstall.sh

# 卸载多实例中的 bi 实例
sudo bash scripts/hop-uninstall.sh --instance bi --yes

# 彻底清理（最后一个实例时才允许）
sudo bash scripts/hop-uninstall.sh --purge --yes
```

## 10. LLM 助手独立部署（可选）

> **定位**：本节是主部署流程的**可选扩展**。Hop Web 的 AI 个人助手（"NL-to-pipeline"自然语言生成管道）依赖两个外部服务：
> - **LiteLLM Proxy**：OpenAI 兼容网关，统一对接阿里云 DashScope（通义千问）
> - **Qdrant**：向量数据库，用于知识库 RAG 检索
>
> 这两个服务**不在主离线包**中，原因有二：
> 1. 它们以 Docker 容器形式运行，与 Hop Web 的主机原生部署模式不同
> 2. 首次部署需要从 Docker Hub / GHCR 拉取镜像（可临时联网），与"完全离线"的主流程冲突
>
> 因此采用**独立部署脚本** `hop-deploy-llm.sh`，与主部署解耦。是否启用 LLM 助手不影响 Hop Web 核心功能。

### 10.1 前置条件

| 项目 | 要求 |
|------|------|
| Docker Engine | 24.0+（含 docker compose v2 插件，或 docker-compose v1） |
| 网络 | 首次部署需临时访问公网（`ghcr.io`、`docker.io`）；运行时 LLM API 调用需访问阿里云 DashScope |
| 阿里云账号 | 已开通 DashScope（模型服务灵积），获取 API Key |
| Hop Web | 已通过 `hop-deploy.sh` 部署完毕（顺序无强约束，但配置集成需在主部署后） |
| 资源 | 额外 2 GB 内存（litellm ~512MB + qdrant ~1GB）、5 GB 磁盘（向量数据） |

### 10.2 架构与网络拓扑

### 10.3 部署步骤

#### 10.3.1 定位 LLM 助手脚本

LLM 助手脚本已包含在主离线包的 `llm/` 子目录中（由 `hop-package.sh` 自动打包），**无需单独传输**。解压主包后即可直接使用：

```bash
# 假设主包已解压到 /opt/deploy/hop-offline-<version>/
ls /opt/deploy/hop-offline-*/llm/
# 预期输出：
#   docker-compose.llm.yml
#   env.example
#   hop-deploy-llm.sh
#   litellm-config.yaml
```

> **文件说明**：
> - `hop-deploy-llm.sh`：主部署脚本（执行入口）
> - `docker-compose.llm.yml`：Docker Compose 模板（litellm + qdrant 服务定义）
> - `litellm-config.yaml`：LiteLLM 路由配置（DashScope 模型映射）
> - `env.example`：`.env` 模板（参考用，实际 `.env` 由脚本交互式生成）

#### 10.3.2 执行部署（交互式，推荐首次使用）

```bash
# 登录目标服务器，进入主包解压目录
ssh user@target-host
cd /opt/deploy/hop-offline-*/llm/

# 执行（交互式引导填写 API Key）
sudo bash ./hop-deploy-llm.sh
```

> **注意**：脚本会自动从自身所在目录查找 `docker-compose.llm.yml` 与 `litellm-config.yaml`，因此**必须在 `llm/` 目录内执行**，或将整个 `llm/` 目录移动到目标位置（如 `/opt/deploy/llm-scripts/`）后执行。

脚本会依次执行：
1. **前置检查**：验证 docker、docker compose 可用，daemon 运行中
2. **凭据收集**：交互式引导输入 DashScope API Key 与 Base URL（或复用既有 `.env`）
3. **目录准备**：创建 `/opt/hop-llm/`，拷贝 compose 模板与 litellm 配置，生成 `.env`（权限 600）
4. **镜像拉取**：`docker compose pull`（需临时联网，拉取 litellm + qdrant 镜像）
5. **启动服务**：`docker compose up -d`，等待健康检查（最长 60s）
6. **输出摘要**：打印端口、配置文件路径、Hop Web 集成所需的环境变量片段

#### 10.3.3 部署后集成 Hop Web

部署脚本会在 `/opt/hop-llm/hop-web-env.sh` 生成 Hop Web 侧需要的环境变量片段。将其追加到 setenv.sh 并重启：

```bash
# 追加 LLM 相关环境变量到 setenv.sh
sudo bash -c 'cat /opt/hop-llm/hop-web-env.sh >> /opt/hop/tomcat-run/bin/setenv.sh'

# 重启 Hop Web 使配置生效
sudo systemctl restart qi-hop-web

# 验证 Hop Web 能访问 LLM 服务
curl -s http://localhost:4000/health/liveliness   # litellm 存活
curl -s http://localhost:6333/healthz              # qdrant 存活
```

### 10.5 参数速查

| 参数 | 说明 | 默认值 |
|------|------|--------|
| `--api-key <sk-xxx>` | DashScope API Key（必填，或交互式输入） | 空 |
| `--api-base <url>` | DashScope API Base URL | `https://dashscope.aliyuncs.com/compatible-mode/v1` |
| `--model <name>` | 默认模型（`qwen-turbo`/`qwen-plus`/`qwen-max`） | `qwen-plus` |
| `--port <n>` | LiteLLM 监听端口 | `4000` |
| `--qdrant-port <n>` | Qdrant HTTP 端口 | `6333` |
| `--base <dir>` | 安装目录 | `/opt/hop-llm` |
| `--skip-pull` | 跳过镜像拉取（已离线导入镜像时使用） | `false` |

### 10.8 常见问题

#### 10.8.1 LLM 助手不工作（Hop Web 日志报连接拒绝）

### 10.9 与主部署的解耦说明

> **关键澄清**：虽然 LLM 助手脚本（`llm/`）随主包分发，但它的**部署、运行、卸载**完全独立于 Hop Web 主部署。两者解耦体现在：
> - **部署独立**：主部署用 `scripts/hop-deploy.sh`，LLM 助手用 `llm/hop-deploy-llm.sh`，互不依赖
> - **运行独立**：Hop Web 是主机原生进程（systemd），LLM 助手是 Docker 容器（docker compose）
> - **网络独立**：主部署完全离线，LLM 助手部署时需临时联网，运行时仅 LLM API 需公网
> - **生命周期独立**：任一方升级/卸载不影响另一方

| 场景 | 主部署（Hop Web） | LLM 助手 | 说明 |
|------|------------------|----------|------|
| 仅核心 ETL 功能 | ✅ 必需 | ❌ 可不部署 | Hop Web 不依赖 LLM 也能正常运行 |
| 启用 AI 助手 | ✅ 必需 | ✅ 必需 | LLM 助手为 Hop Web 提供自然语言生成管道能力 |
| 主部署升级 | 重新执行 `scripts/hop-deploy.sh` | 不受影响 | LLM 栈独立运行，无需重启 |
| LLM 栈升级 | 不受影响 | `docker compose pull && up -d` | Hop Web 无需重启（仅下次调用 LLM 时生效） |
| 主部署卸载 | `scripts/hop-uninstall.sh` | 需单独卸载 | `llm/hop-deploy-llm.sh --uninstall` |
