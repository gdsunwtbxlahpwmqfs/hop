### 1.1 NullPointerException 修复

**问题描述**：右键菜单点击 "Close Others"/"Close tabs to the right"/"Close tabs to the left" 等选项时出现空指针异常：

```
java.lang.NullPointerException: Cannot invoke "org.eclipse.swt.custom.CTabItem.getParent()" because "selectedTabItem" is null
```

**根本原因**：在 Hop Web（RAP）环境中，浏览器到服务器的坐标往返可能导致标签页命中测试失效，使 `selectedTabItem` 为 `null`。

### 1.2 getTabsToRight 方法逻辑修复

**问题描述**：右键菜单点击 "Close tabs to the right" 选项时不起作用。

**根本原因**：`getTabsToRight` 方法的遍历逻辑错误。原代码从右向左遍历，导致无法正确识别"右侧"的标签页。

**修复位置**：[TabClosable.java](file:///Users/yixiao/Documents/a723wlkjgzs_code/mygithub/hop/ui/src/main/java/org/apache/hop/ui/hopgui/perspective/TabClosable.java) 第32-48行

**修复详情**：

```java
// 修复前：从右向左遍历，逻辑错误
default List<CTabItem> getTabsToRight(CTabItem selectedTabItem) {
  if (selectedTabItem == null) {
    return new ArrayList<>();
  }
  CTabFolder folder = selectedTabItem.getParent();
  List<CTabItem> items = new ArrayList<>();
  for (int i = folder.getItems().length - 1; i >= 0; i--) {  // 错误：从右向左遍历
    if (selectedTabItem.equals(folder.getItems()[i])) {
      break;
    } else {
      items.add(folder.getItems()[i]);
    }
  }
  return items;
}

// 修复后：从左向右遍历，找到选中项后收集右侧标签
default List<CTabItem> getTabsToRight(CTabItem selectedTabItem) {
  if (selectedTabItem == null) {
    return new ArrayList<>();
  }
  CTabFolder folder = selectedTabItem.getParent();
  List<CTabItem> items = new ArrayList<>();
  boolean foundSelected = false;
  for (CTabItem item : folder.getItems()) {
    if (foundSelected) {
      items.add(item);  // 收集选中项右侧的所有标签
    }
    if (selectedTabItem.equals(item)) {
      foundSelected = true;  // 标记找到选中项
    }
  }
  return items;
}
```

## 三、启动脚本修复

### 3.1 问题描述

`start-web-dev.sh` 脚本存在一个共性问题：所有模块（core、beam、web、plugins）的提取逻辑都只检查目标目录是否**为空或不存在**，而没有检查源文件（zip/war）是否比目标目录**更新**。这导致重新编译后新的代码无法被部署！

### 3.2 修复内容

修改了 [start-web-dev.sh](file:///Users/yixiao/Documents/a723wlkjgzs_code/mygithub/hop/dev-scripts/start-web-dev.sh) 中的四个模块提取逻辑：

#### 3.2.1 Core Assembly 修复（第11-22行）

```bash
# 修复前：只检查目录是否存在
elif [ ! -d "$CORE_DIR" ]; then

# 修复后：检查目录是否存在 OR zip 文件是否更新
elif [ ! -d "$CORE_DIR" ] || [ "$CORE_ZIP" -nt "$CORE_DIR" ]; then
  rm -rf "assemblies/core/target/hop"  # 先删除旧目录
  unzip -q "$CORE_ZIP" -d "assemblies/core/target/"
```

#### 3.2.2 Beam Assembly 修复（第24-35行）

```bash
# 修复前：只检查目录是否存在
elif [ ! -d "$BEAM_DIR" ]; then

# 修复后：检查目录是否存在 OR zip 文件是否更新
elif [ ! -d "$BEAM_DIR" ] || [ "$BEAM_ZIP" -nt "$BEAM_DIR" ]; then
  rm -rf "assemblies/beam/target/hop"  # 先删除旧目录
  unzip -q "$BEAM_ZIP" -d "assemblies/beam/target/"
```

#### 3.2.3 Webapp 修复（第37-47行）

```bash
# 修复前：只检查目录是否为空
if [ -z "$(ls -A assemblies/web/target/webapp 2>/dev/null)" ]; then

# 修复后：检查目录是否为空 OR war 文件是否更新
WEBAPP_DIR="assemblies/web/target/webapp"
if [ -z "$(ls -A "$WEBAPP_DIR" 2>/dev/null)" ] || [ "assemblies/web/target/hop.war" -nt "$WEBAPP_DIR" ]; then
  rm -rf "$WEBAPP_DIR"  # 先删除旧目录
  unzip -q assemblies/web/target/hop.war -d "$WEBAPP_DIR"
```

#### 3.2.4 Plugins 修复（第62-72行）

```bash
# 修复前：只检查目录是否存在
if [ ! -d "$PLUGINS_DIR/transforms" ] && [ -f "$PLUGINS_ZIP" ]; then

# 修复后：检查目录是否存在 OR zip 文件是否更新
if [ -f "$PLUGINS_ZIP" ] && ([ ! -d "$PLUGINS_DIR/transforms" ] || [ "$PLUGINS_ZIP" -nt "$PLUGINS_DIR" ]); then
  rm -rf "$PLUGINS_DIR"  # 先删除旧目录
  # ... 提取逻辑
```

### 3.3 修复逻辑

| 条件 | 行为 |
|------|------|
| 目标目录为空或不存在 | 提取源文件 |
| 源文件（zip/war）比目标目录新 | **重新提取**源文件 |
| 两者都不满足 | 使用已有的目标目录（增量优化） |

### 3.4 修改思路总结

#### 核心问题
脚本的原始逻辑存在"一次提取，永久有效"的缺陷，无法响应源文件的更新。

#### 解决策略
1. **时间戳对比**：使用 `-nt` 操作符检查源文件是否比目标目录新
2. **先删后建**：提取前先删除旧目录，确保完全替换
3. **逻辑重构**：将提取条件从单一条件改为复合条件（OR 连接）

#### 修复模式
```bash
# 通用修复模式
SOURCE_FILE="path/to/source.zip"
TARGET_DIR="path/to/target"

if [ ! -d "$TARGET_DIR" ] || [ "$SOURCE_FILE" -nt "$TARGET_DIR" ]; then
  rm -rf "$TARGET_DIR"    # 删除旧目录
  mkdir -p "$TARGET_DIR"  # 创建新目录
  unzip -q "$SOURCE_FILE" -d "$TARGET_DIR"  # 提取文件
fi
```

#### 应用范围
此修复模式适用于脚本中所有从压缩文件提取内容的场景：
- Core Assembly (hop-core-assembly-*.zip)
- Beam Assembly (hop-beam-assembly-*.zip)  
- Webapp (hop.war)
- Plugins (hop-assemblies-plugins-*.zip)

#### 收益
- **开发效率提升**：修改代码后重新编译，启动脚本会自动使用新代码
- **避免手动清理**：无需手动删除旧目录
- **增量优化保留**：未更新时仍使用缓存，加快启动速度

## 七、常见问题排查

### 7.1 NullPointerException 再次出现

**问题现象**：右键菜单点击 "Close Others" 等选项时出现 NPE。

**排查步骤**：

1. 检查 [TabClosable.java](file:///Users/yixiao/Documents/a723wlkjgzs_code/mygithub/hop/ui/src/main/java/org/apache/hop/ui/hopgui/perspective/TabClosable.java) 中的三个方法是否都有空值检查；
2. 检查 [TabCloseHandler.java](file:///Users/yixiao/Documents/a723wlkjgzs_code/mygithub/hop/ui/src/main/java/org/apache/hop/ui/hopgui/perspective/TabCloseHandler.java) 中的 `selectedItem` 是否正确初始化；
3. 确认已重新编译并部署最新代码。

### 7.2 国际化不生效

**问题现象**：菜单或对话框文本显示英文而非中文。

**排查步骤**：

1. 检查资源文件（`messages_zh_CN.properties`）是否添加了对应的键值对；
2. 检查代码中的键名是否与资源文件中的一致；
3. 确认使用了正确的 `PKG` 类进行国际化查找；
4. 检查系统语言设置是否正确。

### 7.3 编译失败

**问题现象**：Maven 编译报错。

**排查步骤**：

```bash
# 清理并重新编译
./mvnw clean install -pl ui -am -DskipTests

# 检查依赖问题
./mvnw dependency:tree -pl ui

# 检查日志中的错误信息
./mvnw install -pl ui -am -DskipTests 2>&1 | grep -A 10 "error"
```

### 7.4 服务器启动失败

**问题现象**：Docker 容器无法启动。

**排查步骤**：

```bash
# 检查容器日志
docker logs hop-web-dev

# 检查端口是否被占用
lsof -i :8080

# 重新构建并启动
./mvnw clean install -pl assemblies/web -am -DskipTests -Dfast-build
./dev-scripts/start-web-dev.sh
```
