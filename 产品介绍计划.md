## 一、项目背景与目标

### 1.1 任务
为 Qi Hop 编写一份面向**业务管理者**的产品介绍文档（Markdown，约 1500-2000 字），突出产品功能特点与差异化优势，配以**全新截图**（不引用历史截图）。

### 1.2 产品定位
Qi Hop 是基于 Apache Hop（版本 `2.19.0-SNAPSHOT`，Apache 2.0 协议）的数据/元数据编排平台（ETL）。在原生 Hop 基础上，Qi Hop 增值了 **AI 智能化、内置终端、技能管理系统、中文本地化** 等能力。

### 1.3 受众与文风
- **读者**：业务管理者（CTO/技术总监/架构师）
- **文风**：弱化技术细节（不写 Maven 模块、JVM 参数），强调业务价值（降本增效、降低门槛），多用对比和量化
- **重点**：突出 Qi Hop 相对原生 Apache Hop 的增值能力

## 二、调研结论

### 2.1 核心特性矩阵

| 特性 | 说明 | Qi Hop 增值点 |
|------|------|--------------|
| 可视化 Pipeline 设计 | 拖拽式画布、所见即所得、285+ 插件 | 双端（RCP 桌面 + RAP Web）|
| 元数据驱动架构 | 配置优于代码、可视化审计 | — |
| 多引擎执行 | Native / Beam / Spark / Flink / Dataflow | — |
| **LLM AI 助手** | 自然语言对话、RAG 知识库 | ✅ 原生 Hop 无此能力 |
| **Skills 技能管理** | GLOBAL/MANUAL 双触发、团队共享 | ✅ 原生 Hop 无此能力 |
| **内置 Terminal** | 浏览器内直接执行命令、多 Tab | ✅ 原生 Hop 无此能力 |
| 中文文档/本地化 | 完整中文助手手册、URL 映射 | ✅ 原生 Hop 无此能力 |

### 2.2 四大特色功能 UI 入口（已通过代码调研确认）

| 功能 | UI 入口 | 定位方式 | 前置条件 |
|------|---------|---------|---------|
| **Pipeline 画布** | 文件树双击 `.hpl` / URL 参数 `?file=` | 左侧文件树 / URL 直达 | 无 |
| **LLM AI 助手** | 右下角悬浮按钮（`llm-assistant-fab`） | SVG data key | `HOP_LLM_ENABLED=true` |
| **Skills 管理** | 左侧侧边栏底部 Skills 按钮 | Widget ID `HopGui-SidebarToolbar-Skills` | LLM 启用 |
| **Terminal** | 左侧侧边栏底部 Terminal 按钮 / `Tools→Terminal`（Ctrl+J） | Widget ID / 菜单 | 无 |

### 2.3 关键凭据与直达 URL
- 登录：`admin` / `admin@2026`
- URL 直达参数：`?user=admin&pass=admin@2026&project=samples&file=pipelines/read-transactions.hpl`
- LLM dev 默认模型：`qwen-plus`（通过 LiteLLM 代理）

### 4.3 Playwright 脚本技术要点
- RAP 是 SWT 转 Web，DOM 定位需用 Widget setData 标记
- 每步操作后 `wait_for_timeout(2000-3000)` 让 SWT 重绘
- 使用 `page.wait_for_load_state('networkidle')` 等待 AJAX
- xterm.js 是 canvas，直接整页截图
- 元素定位优先用 `aria-label` 或文字选择器

## 七、风险与应对

| 风险 | 应对 |
|------|------|
| Hop Web 启动失败 | 检查 catalina.out 日志，必要时 `--restart` |
| RAP 应用 DOM 定位困难 | 充分等待渲染（3s+），用整页截图兜底 |
